# Place2Place 本番デプロイ・データベース運用マニュアル

このドキュメントは、**Place2Place** WebアプリケーションをVercelにデプロイし、本番環境のデータベース（Neon Postgres）をクリーンな状態で稼働・維持するための運用・デプロイ手順書です。

---

## 1. データベース分離方針（開発用と本番用）

本アプリは講義やワークショップごとのリセットを行わず、**長期的に回答データを蓄積し続ける設計**となっています。
開発時のテストデータ（モックデータなど）が本番環境の可視化マップに混入するのを避けるため、必ず**開発用DB**と**本番用DB**を論理的または物理的に分離します。

- **開発用DB (現在の接続先)**: ローカルでの検証、モックテスト投稿、クォータ切れ時の動作テスト等に使用。
- **本番用DB (新規作成)**: ワークショップ等の実稼働時に使用。初期状態ではエリアのシードデータ20件のみが存在し、回答データは空の状態で運用を開始します。

---

## 2. 本番用データベースの作成手順 (Neon)

本番用データベースをNeon上に用意するには、以下のいずれかの方法を行います。

### 方法A: 同一Neonプロジェクト内で新規データベースを作成する (推奨)
1. Neonコンソールにログインし、プロジェクトページを開きます。
2. 左メニューの **Databases** を選択し、**Create Database** ボタンをクリックします。
3. データベース名（例: `place2place_prod`）を入力して作成します。
4. **Dashboard** に戻り、接続文字列（Connection String）のデータベース部分を新しく作成したものに変更した URL を取得します。
   - 例: `postgresql://user:password@host/place2place_prod?sslmode=require`

### 方法B: 本番専用の別Neonプロジェクトを新規作成する
1. Neonコンソールで **Create Project** を行い、本番専用のプロジェクト（例: `place2place-prod`）を作成します。
2. 作成された新しいプロジェクトの `DATABASE_URL` を取得します。

---

## 3. Vercel 本番環境への環境変数登録

Vercelの管理画面（Project Settings > Environment Variables）を開き、以下の環境変数を設定します。

| 変数名 | 設定内容 | 備考 |
| :--- | :--- | :--- |
| `DATABASE_URL` | **本番用データベース**の接続文字列 | 上記で取得した本番用DBのURL |
| `OPENAI_API_KEY` | 実APIキー | クレジット残高のある有効なOpenAI APIキー |
| `UPSTASH_REDIS_REST_URL` | Upstash RedisのREST URL | レートリミット管理用。本番では必須 |
| `UPSTASH_REDIS_REST_TOKEN` | Upstash RedisのREST Token | レートリミット管理用。本番では必須 |
| `SUBMISSIONS_ENABLED` | `true` または `false` | `false` に設定すると新規投稿を緊急停止（デフォルト: `true`） |

> [!IMPORTANT]
> - Vercel本番環境には **`MOCK_EMBEDDINGS` 環境変数を絶対に登録しないでください**。未登録（未定義）の状態にすることで、自動的に実OpenAI APIを用いたEmbedding生成が実行されます。

## 3.5. レートリミット（Upstash Redis）および投稿制限の設定

### ① Upstash Redis データベースの作成方法
1. [Upstash Console](https://console.upstash.com/) にサインアップ・ログインします。
2. **Create Database** ボタンをクリックし、以下の設定でデータベースを作成します。
   - **Name**: `place2place-ratelimit` など任意の名前
   - **Type**: `Regional`
   - **Region**: アプリケーションをホストするVercelリージョン（例: `ap-northeast-1` / Tokyo など）に近いものを選択
3. 作成完了後、データベース詳細画面の **REST API** セクションから以下の情報を取得します。
   - `UPSTASH_REDIS_REST_URL`
   - `UPSTASH_REDIS_REST_TOKEN`

### ② Vercelへの設定
Vercelのプロジェクト設定画面（Project Settings > Environment Variables）にて、上記で取得した `UPSTASH_REDIS_REST_URL` および `UPSTASH_REDIS_REST_TOKEN` を環境変数として登録します。

> [!IMPORTANT]
> **本番環境（production）での未設定時の挙動**:
> - 本番環境において `UPSTASH_REDIS_REST_URL` または `UPSTASH_REDIS_REST_TOKEN` のいずれかが設定されていない場合、セキュリティとシステム保護のため、新規の投稿（回答の送信）はサーバー側で**すべて自動的に拒否**されます。
> - 一方、ローカル開発環境（`NODE_ENV !== 'production'`）では、環境変数が未設定の場合に警告ログを出力した上でレートリミット判定をバイパスし、投稿自体は成功します。

### ③ 緊急停止スイッチ（`SUBMISSIONS_ENABLED`）による投稿停止
- 本番環境またはローカル開発環境で、何らかの理由により新規投稿の受付を一時的に停止したい場合、環境変数 `SUBMISSIONS_ENABLED` に `false` を設定します。
- `SUBMISSIONS_ENABLED=false` が検出された場合、サーバーアクション側で投稿処理が即座に拒否され、OpenAI APIの呼び出しも実行されません。

## 4. 本番データベースの初期セットアップ手順

デプロイメント（Vercelのビルド）を行う前に、ローカル環境から**本番データベース**に対してpgvector拡張の有効化、テーブル定義の作成、初期エリアデータのシード投入を行います。

以下の順番で実行してください。

### ① pgvector拡張の有効化
本番用DBでベクトル検索を行うために、pgvector拡張機能を有効化します。このスクリプトは安全で冪等（何回実行しても安全）です。
```bash
DATABASE_URL="[本番用の接続文字列]" npm run db:enable-vector
```

### ② スキーママイグレーション (テーブル作成)
Drizzle Kitを用いて本番環境のデータベースにテーブルスキーマを反映（作成）します。
```bash
DATABASE_URL="[本番用の接続文字列]" npm run db:push
```

### ③ エリアシードデータの投入
本番用DBに対して、東京の初期20エリアを登録します。
```bash
DATABASE_URL="[本番用の接続文字列]" npm run db:seed
```
※ このスクリプトは `ON CONFLICT DO NOTHING` 設計（差分追加のみ）となっているため、すでにエリアが存在する場合も、既存の回答データやエリア情報を削除・上書きすることは絶対にありません。複数回実行しても安全です。

## 5. Vercel デプロイと動作確認

1. 環境変数の設定およびデータベースのセットアップ完了後、Vercelでデプロイ（**Deploy**）を実行します。
2. デプロイ完了後、公開されたドメインにアクセスします。
3. **初期表示の確認**:
   - トップページに「お題のエリア」がランダムに提示されていること。
   - 2Dプロット部分に「回答が集まるとここにプロットされます」と表示されていること。
   - 画面下部の「回答待ちのエリア」に、シードされた20個の街の名前が五十音順チップで並んでいること。
4. **ポーリングの確認**:
   - `/api/visualization` にブラウザ等でアクセスし、回答数 `0` の状態で空の `plotted` 配列と20エリアの `unplotted` 配列が含まれた JSON が返却されること。

## 6. 長期データ蓄積型運用のガイドライン

Place2Placeの継続的な運用のために、以下の性質を理解し運用してください。

### データの永続性について
- 本アプリはデータ蓄積型であり、講義やワークショップの回ごとに自動的あるいは手動でデータベースをクリアする機能はMVP段階では存在しません。
- 過去のワークショップで集まった記述データもすべて永続化され、次回以降のプロットに累積平均として影響し続けます。

### 本番環境でのテスト投稿の制限
- 本番用ドメインで動作確認の目的でテスト投稿を行うと、そのテスト投稿も永続データとして蓄積され、プロットの計算対象に入ってしまいます。
- デモ前の動作チェックや疎通テストは、極力**開発用DB**と接続したローカル環境（`npm run dev`）で完結させてください。本番環境でテスト投稿を行う場合は、最小限（1〜2件程度）に留めてください。

### 将来的なロードマップ（イベント/ワークショップ単位の分離）
将来的に「講義A」と「講義B」で回答データを分離し、それぞれ独立したプロットを表示したくなった場合は、以下の拡張を検討してください。
1. `events` テーブル（`id`, `name`, `passcode`, `created_at`）を作成。
2. `responses` テーブルおよび `area_embeddings` テーブルに `event_id` カラムを追加し、イベントごとの複合ユニークキーでベクトル加算集計を分離。
3. フロントエンドのURLにクエリパラメータ（例: `?event=class_a`）を追加してフィルタリング。

## 7. メンテナンス・トラブルシューティング

### area_embeddings の再構築 (集計不整合からの復旧)
投稿処理におけるNeon HTTPドライバの制約（トランザクションが使えない逐次書き込み）により、万一 `responses` 保存直後に `area_embeddings` の加算更新処理で失敗した場合、集計値にズレが生じる可能性があります。
もし可視化データの不整合（回答数やプロット座標の不一致など）が疑われる場合は、以下のメンテナンススクリプトを実行して、`responses` テーブルに保存されているすべての生データから `area_embeddings` テーブルの累積値（ベクトル和と回答数カウント）を正確に再構築できます。

```bash
DATABASE_URL="[本番用の接続文字列]" npm run db:rebuild-area-embeddings
```

- **安全設計のポイント**:
  - `responses` テーブルおよび `areas` テーブルに対する削除・変更は一切行いません。
  - `area_embeddings` レコードを安全に upsert します。
  - 回答本文（`description`）を出力することはありません（セキュリティ設計の維持）。
  - 集計処理時に、DB上に対応する `responses` レコードが存在しない `area_id` が `area_embeddings` に見つかった場合、誤削除を防ぐためレコードの削除は行わず、警告（WARNING）ログのみを表示します。
- このスクリプトは通常運用時には実行不要です。あくまで不整合が発生した疑いがある場合のメンテナンス用としてご利用ください。

### デバッグ用APIの利用制限 (?debug=1)
- 可視化データの類似度行列や距離行列、MDS計算結果の中間状態を検証するために `/api/visualization?debug=1` というデバッグパラメータが用意されています。
- このデバッグ情報はローカル開発環境（`NODE_ENV !== 'production'`）では自動的に有効になりますが、本番のVercel環境では外部へのデータ露出防止のため、デフォルトで無効化（`debug` オブジェクトを返さない設計）になっています。
- もし本番環境で検証の目的で一時的にデバッグ表示を許可したい場合は、Vercelの環境変数に以下を登録してください：
  * `VISUALIZATION_DEBUG`: `true`
  * ※この環境変数が設定されない限り、本番環境で `?debug=1` を付与してもデバッグ情報は一切出力されません。
  * ※また、デバッグ情報が有効であっても、個人情報の回答本文（`description`）、DB認証情報、APIキー等は一切出力されない安全な設計となっています。
